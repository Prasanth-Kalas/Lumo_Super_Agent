# ADR-016 — Agent Cost Metering and Budgets

**Status:** Proposed (drafted 2026-04-27, Phase 4). Codex COST-1 implements against this ADR.
**Authors:** Coworker M (architecture pass), to be reviewed by Kalas (CEO/CTO/CFO).
**Related:** `docs/specs/adr-013-agent-runtime-contract.md` (cost ceiling at the contract layer),
`docs/specs/adr-014-agent-permissions-capabilities.md` (spending caps via scopes),
`docs/specs/adr-015-marketplace-distribution-trust-tiers.md`,
`docs/specs/phase-4-outlook.md`,
`docs/specs/phase-3-master.md` (RUNTIME-1's forecasting precedent).
**Implements:** the per-invocation cost log, the per-user budget, the
per-agent ceiling, the cost-routing fallback, and the user-facing
reporting surface.

---

## 1. Context

A marketplace of agents is a cost-amplification surface. One agent
that calls Brain tools 50 times per invocation can chew through the
< $150/mo target single-handedly. The platform has to meter every
invocation, enforce both per-user and per-agent ceilings, and
gracefully degrade to cheaper paths when ceilings approach.

ADR-016 is the cost contract. It sits below ADR-013 (which defines
the manifest's cost-model) and below ADR-014 (which defines spending
caps as scope qualifiers). Together they make a defence-in-depth: the
manifest declares an upper bound, the user grants a per-action cap,
and the platform enforces a per-user budget on top.

Phase 4's posture matches Phase 3: < $150/mo at 1k MAU; self-host
before paid services. The cost log is the source of truth for both
the user-facing budget UI and the platform's own runtime-intelligence
analytics (RUNTIME-1's per-call forecasting in Phase 3).

## 2. Per-invocation accounting

Every agent invocation writes one row to `agent_cost_log`:

```sql
create table public.agent_cost_log (
  id                    bigint generated by default as identity primary key,
  user_id               uuid not null references public.profiles(id) on delete cascade,
  agent_id              text not null,
  agent_version         text not null,
  mission_id            uuid references public.missions(id) on delete set null,
  mission_step_id       uuid,
  request_id            uuid not null,                  -- ties to ADR-013 retry semantics
  brain_calls_usd       numeric(10,6) not null default 0,
  brain_call_count      integer not null default 0,
  model_tokens_input    integer not null default 0,
  model_tokens_output   integer not null default 0,
  model_tokens_cost_usd numeric(10,6) not null default 0,
  connector_calls       integer not null default 0,
  connector_calls_usd   numeric(10,6) not null default 0,
  total_usd             numeric(10,6) not null default 0,
  status                text not null check (status in ('completed','aborted_budget','aborted_error')),
  created_at            timestamptz not null default now()
);

create index agent_cost_log_by_user_day
  on public.agent_cost_log (user_id, created_at desc);

create index agent_cost_log_by_user_agent_day
  on public.agent_cost_log (user_id, agent_id, created_at desc);

create index agent_cost_log_by_agent_day
  on public.agent_cost_log (agent_id, created_at desc);
```

The row is written by the orchestrator at the end of each invocation
(success or failure). Retries on the same `request_id` UPDATE the
row in place rather than appending — the cost log tracks
*invocations*, not *attempts*. Aborted-mid-flight invocations (sandbox
timeout, OOM, kill) write a row with `status = 'aborted_error'` and
the partial cost incurred up to abort.

The `total_usd` is the sum of `brain_calls_usd + model_tokens_cost_usd
+ connector_calls_usd + per_invocation_flat_usd` (the last is from
the manifest's `cost_model.per_invocation_usd` if declared). It is
denormalised for fast dashboard queries.

## 3. Per-user budgets

Three tiers in v1; the data model accommodates Phase 5 expansion:

```sql
create table public.user_budget_tier (
  user_id      uuid primary key references public.profiles(id) on delete cascade,
  tier         text not null check (tier in ('free','pro','enterprise')),
  daily_cap_usd  numeric(10,2) not null,
  monthly_cap_usd numeric(10,2) not null,
  effective_from timestamptz not null default now(),
  notes        text
);
```

| Tier | Daily cap | Monthly cap | Notes |
|---|---|---|---|
| `free` | $0.50 | $5.00 | Default for new users |
| `pro` | $5.00 | $50.00 | Subscription, Phase 5 |
| `enterprise` | metered | metered | Per-contract; soft-cap with billing |

### 3.1 Enforcement at dispatch

Before every invocation, the orchestrator computes:

```
spent_today_usd  = sum(agent_cost_log.total_usd) where user_id = U and created_at >= today_start
spent_month_usd  = sum(agent_cost_log.total_usd) where user_id = U and created_at >= month_start
remaining_daily  = daily_cap_usd  - spent_today_usd
remaining_monthly = monthly_cap_usd - spent_month_usd
```

If `remaining_daily <= 0` or `remaining_monthly <= 0`:

- The orchestrator refuses dispatch with `BUDGET_EXCEEDED`.
- The user sees: "You've hit your daily limit for agent activity.
  Resets at midnight, or upgrade to Pro for higher limits." (or the
  monthly equivalent).
- The mission step transitions to `failed` with `error_text =
  'budget_exceeded'`.
- An alert email fires if this is the first breach of the period.

### 3.2 Pre-flight forecast

The platform forecasts the invocation's cost before dispatch, using
the manifest's `cost_model.max_cost_usd_per_invocation` as the upper
bound. If `max_cost_usd_per_invocation > remaining_daily`, the
orchestrator refuses dispatch even before the agent runs — better to
fail fast than to start a workload that will be aborted mid-flight.

Mid-flight budget breaches (the agent's actual spend exceeds
`remaining_daily` halfway through) are detected by the connector
dispatcher and Brain SDK gateway: each call subtracts from the budget
in-memory, and a call that would push spend over the cap is refused
with `BUDGET_EXCEEDED`. The agent receives the typed error and
returns `status: failed`.

### 3.3 Per-user grace

A user who hits the daily cap may grant themselves a one-time
"continue for $X more" via an in-app prompt, charged immediately to
their card on file (Pro) or held against their enterprise meter
(Enterprise). Free-tier users see no grace prompt — they wait for
midnight or upgrade.

## 4. Per-agent budgets (developer-imposed)

The manifest's `cost_model.max_cost_usd_per_invocation` is a
**developer-declared self-ceiling**. The platform enforces it
server-side per ADR-013 invariant 6 — the agent is never trusted to
honour its own ceiling.

Enforcement:

- Connector dispatcher and Brain SDK gateway refuse calls that
  would push the running invocation total over
  `max_cost_usd_per_invocation`.
- The refused call returns `BUDGET_EXCEEDED`; the agent surfaces
  this to the user.
- The cost log row carries `status = 'aborted_budget'` if the
  invocation aborted because of the per-agent ceiling.

This protects users from runaway agents. A bug in `summarize-emails-daily`
that loops on 10 000 Brain calls hits the manifest ceiling first
(say $0.04) and aborts, regardless of the user's budget.

## 5. Cost-routing decisions

When a Brain SDK call would exceed the remaining budget but a cheaper
model is available, the SDK falls back to the cheaper model rather
than aborting. This logic is in the Brain SDK, not in agents:

| Original model | Fallback model | When triggered |
|---|---|---|
| `claude-opus-4` | `claude-haiku-4` | `remaining_user_daily_usd < forecast_with_opus` |
| `claude-haiku-4` | (no fallback; abort) | Below haiku is "no model" |
| `gpt-5` | `gpt-5-mini` | Same condition |
| `gemini-pro` | `gemini-flash` | Same condition |

The fallback is logged in `agent_cost_log.evidence` (Codex extends
the schema with a small jsonb) so the user can see "used cheaper
model" in their cost report. Fallback applies even when the
per-agent ceiling is tight; the SDK always prefers a cheaper-but-still-usable
result over a "sorry, can't run."

The cost-routing decisions defer to RUNTIME-1's forecasting (Phase 3)
where available. RUNTIME-1's `lumo_forecast_metric` extension to cost
prediction is the input here. When RUNTIME-1's forecast confidence is
low (< 0.6), the SDK uses the manifest's declared `per_token_usd`
rates as a static fall-back forecast.

## 6. Reporting

### 6.1 In-app

Workspace → Settings → Agents → [Agent] surfaces:

- This-month-cost (running total).
- Daily breakdown for the last 30 days (sparkline).
- Per-capability breakdown.
- "Forecast for the rest of the month" (linear extrapolation).

Workspace → Settings → Budget surfaces:

- Tier, daily cap, monthly cap.
- Spend today / this month.
- Top 5 agents by spend, this month.
- Upgrade CTA (free → pro).

### 6.2 Daily email digest

Every morning at the user's local 7am, an email goes out if the user
spent > $0.01 the previous day:

> Yesterday Lumo agents cost $0.34. Top 3:
>
> 1. Daily Email Summary — $0.18 (5 invocations)
> 2. Trip Planner — $0.12 (1 invocation)
> 3. Weather Now — $0.04 (12 invocations)
>
> Daily limit: $0.50 (you used 68%). [View details]

Free-tier users see a nudge to upgrade if they hit > 80% of the
daily cap on > 3 days in a 7-day window.

### 6.3 Monthly summary

First of every month, an email summary covering the prior month:

- Total spend, vs. previous month.
- Per-agent cost breakdown.
- Per-capability breakdown.
- "You saved $X by using cheaper-model fallback Y times."
- Action recommendations: agents that cost a lot but have low
  invocation count are flagged ("consider uninstalling").

### 6.4 CSV export

Workspace → Settings → Budget → Export. Downloads
`agent-cost-<user_id>-<from>-<to>.csv` with one row per invocation:
`created_at, agent_id, agent_version, mission_id, capability_id,
total_usd, brain_calls_usd, model_tokens_input, model_tokens_output,
connector_calls, status`.

## 7. Free-tier specific posture

Free is the default. The cost shape at 1k MAU determines whether the
platform is solvent.

### 7.1 Targets

- Median free user spend: $0.10/day (≈ $3/month).
- P95 free user spend: $0.40/day (≈ $12/month — below cap).
- 95% of invocations under $0.005 (a single small-Brain call cost).
- 99% of invocations under $0.05 (the manifest median ceiling).

### 7.2 Acceptable degradation under load

If platform-wide spend trends toward $150/mo at 1k MAU:

1. The Brain SDK aggressively prefers the cheapest model in its
   model-routing decisions for free-tier users.
2. The orchestrator refuses dispatching `experimental`-tier agents
   for free-tier users for the rest of the day if the user has hit
   60% of cap.
3. Pro/Enterprise users see no degradation — they're paying for
   the consistency.

This is a graceful-degradation policy, not a cliff. The user is
still fundamentally able to use Lumo; they just trade quality for
cost on the margin.

## 8. Acceptance criteria for COST-1

1. Migration adds `agent_cost_log` and `user_budget_tier` tables
   with documented indexes.
2. Cost row written for every agent invocation; CI integration test
   exercises a 5-step mission with 3 agents and asserts exactly 3
   cost rows with consistent `total_usd`.
3. Per-user budget enforcement test: a free user with $0.50/day cap
   is refused on the 11th invocation of an agent costing $0.05
   each.
4. Per-agent ceiling test: an agent declaring
   `max_cost_usd_per_invocation: 0.04` that attempts $0.05 worth of
   Brain calls is aborted with `aborted_budget` status.
5. Cost-routing fallback test: the same agent under tight budget
   uses the cheaper model and the cost log records the fallback
   evidence.
6. Daily digest cron runs at 7am local user time; sends email if
   spend > $0.01 the prior day; CI test verifies both content and
   timing.
7. Monthly summary cron runs on the 1st; verifies content for a
   seeded test month.
8. CSV export test: downloads correctly for a 30-day window,
   schema matches §6.4.
9. Budget UI at Workspace → Settings → Budget renders correctly for
   free, pro, and enterprise tier users.

## 9. Consequences

### 9.1 Positive

- Every invocation is metered. No surprise bills.
- Defence-in-depth: manifest ceiling, per-user budget, cost-routing
  fallback. Three independent layers; one failing doesn't compound.
- The cost log feeds RUNTIME-1's forecasting (Phase 3) and BANDIT-1's
  reward signal (Phase 3) — same substrate, multiple readers.
- Free tier is genuinely free. The user understands the limit, sees
  it on every invocation, and can upgrade in one click.

### 9.2 Negative

- Cost log writes add latency to every invocation. Mitigation: write
  is asynchronous (fire-and-forget queue) with a hard sync at
  invocation end; budget check reads from a 5-second cache.
- Mid-flight budget breach gives the user a partial result. The agent
  has to handle this gracefully. Mitigation: the SDK's
  budget-aware Brain client raises a typed error that authors are
  required to catch (lint rule: no un-caught Brain calls).
- Cost-routing fallback can degrade quality silently. Mitigation:
  fallback is logged and visible in the cost report; users can see
  when it kicked in.

### 9.3 Trade-offs accepted

- 5-second budget cache TTL means up to 5 seconds of overspend in
  the worst case (multiple concurrent invocations). At median agent
  cost of < $0.01, the worst-case overspend per user is bounded at
  cents. Acceptable.
- Free-tier $0.50/day is conservative; some users will hit it on
  legitimate use. The grace prompt + Pro upgrade is the answer.

## 10. Alternatives considered

### Option (A) — Pre-paid credits per user
Pro: simple. Con: friction at signup; users hate funding accounts.
Rejected for free-tier; Phase 5 may offer it for Pro.

### Option (B) — No per-user cap, only per-agent
Pro: simpler. Con: a user installing 20 free agents could rack up
real cost without seeing it. Rejected.

### Option (C) — Per-user cap + per-agent ceiling + cost-routing
fallback (CHOSEN)
Pro: the defence-in-depth shape. Con: most surface to maintain.
Acceptable.

### Option (D) — Hard quota (no fallback, abort on cap)
Pro: simple, predictable. Con: brutal user experience — your agent
just stops mid-task. Rejected. Cost-routing fallback is the kindness.

## 11. Open questions

1. **Per-tenant aggregation for Enterprise.** Does the meter
   roll up to the org or stay per-user? **Recommended:** roll up to
   the org for billing; surface per-user breakdown for the admin.
2. **Charity / non-profit tier.** Should we offer a higher daily cap
   for verified non-profits? **Recommended:** Phase 5; out of scope
   for v1.
3. **Cost ceiling on Brain tools themselves.** Some Brain tools
   (e.g., MMRAG with the cross-encoder re-ranker) are expensive;
   should they have their own ceilings independent of agents?
   **Recommended:** Brain side already has rate-limits; agent side
   ceiling is the binding constraint. Revisit if a single Brain tool
   trends toward dominating cost.
